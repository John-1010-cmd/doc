行为型设计模式关注对象之间的通信和职责分配，描述了对象之间如何协作完成单个对象无法完成的任务。本文深入分析七种行为型模式，结合 Spring 源码和实际业务场景，帮助开发者在日常开发中灵活运用。

## 策略模式

### 概念

策略模式定义一系列算法，将每个算法封装到具有公共接口的独立类中，使它们可以互相替换。策略模式让算法的变化独立于使用算法的客户端。

### UML 类图

```
+------------------+        +------------------+
|   Context        |        |   Strategy       |
+------------------+        +------------------+
| - strategy       |------->| + algorithm()    |
| + execute()      |        +------------------+
+------------------+                 ^
                                     |
                    +----------------+----------------+
                    |                |                |
            +-------------+  +-------------+  +-------------+
            | StrategyA   |  | StrategyB   |  | StrategyC   |
            +-------------+  +-------------+  +-------------+
            | + algorithm()|  | + algorithm()|  | + algorithm()|
            +-------------+  +-------------+  +-------------+
```

### 实战场景：支付方式选择

```java
// 策略接口
public interface PaymentStrategy {
    PayResult pay(PayOrder order);
    boolean supports(PayType type);
}

// 微信支付策略
@Component
public class WechatPayStrategy implements PaymentStrategy {
    @Override
    public PayResult pay(PayOrder order) {
        System.out.println("调用微信支付 API，订单号: " + order.getOrderId());
        // 调用微信支付 SDK...
        return PayResult.success("微信支付成功");
    }

    @Override
    public boolean supports(PayType type) {
        return type == PayType.WECHAT;
    }
}

// 支付宝支付策略
@Component
public class AlipayPayStrategy implements PaymentStrategy {
    @Override
    public PayResult pay(PayOrder order) {
        System.out.println("调用支付宝 API，订单号: " + order.getOrderId());
        return PayResult.success("支付宝支付成功");
    }

    @Override
    public boolean supports(PayType type) {
        return type == PayType.ALIPAY;
    }
}

// 银行卡支付策略
@Component
public class BankCardPayStrategy implements PaymentStrategy {
    @Override
    public PayResult pay(PayOrder order) {
        System.out.println("调用银联 API，订单号: " + order.getOrderId());
        return PayResult.success("银行卡支付成功");
    }

    @Override
    public boolean supports(PayType type) {
        return type == PayType.BANK_CARD;
    }
}
```

通过 Spring 自动收集所有策略实现，消除大量 `if-else`：

```java
@Service
public class PaymentService {
    private final Map<PayType, PaymentStrategy> strategyMap;

    // Spring 自动注入所有 PaymentStrategy 实现
    public PaymentService(List<PaymentStrategy> strategies) {
        this.strategyMap = strategies.stream()
                .collect(Collectors.toMap(
                    s -> {
                        // 通过 supports 方法找到对应的 PayType
                        for (PayType type : PayType.values()) {
                            if (s.supports(type)) return type;
                        }
                        throw new IllegalStateException("策略未关联 PayType: " + s.getClass());
                    },
                    Function.identity()
                ));
    }

    public PayResult pay(PayOrder order) {
        PaymentStrategy strategy = strategyMap.get(order.getPayType());
        if (strategy == null) {
            throw new IllegalArgumentException("不支持的支付方式: " + order.getPayType());
        }
        return strategy.pay(order);
    }
}
```

### Spring Resource 加载

Spring 的 `ResourceLoader` 也是策略模式的体现，根据资源位置前缀选择不同的加载策略：

```java
ResourceLoader loader = new DefaultResourceLoader();

// classpath: 前缀 -> ClassPathResource
Resource r1 = loader.getResource("classpath:config.properties");

// file: 前缀 -> FileSystemResource
Resource r2 = loader.getResource("file:/etc/app/config.properties");

// https: 前缀 -> UrlResource
Resource r3 = loader.getResource("https://example.com/config.properties");

// 无前缀 -> 根据 ApplicationContext 类型决定
Resource r4 = loader.getResource("config.properties");
```

## 模板方法模式

### 概念

模板方法在父类中定义算法骨架，将某些步骤延迟到子类实现。子类可以在不改变算法结构的情况下重新定义某些步骤。

### UML 类图

```
+-----------------------+
|   AbstractClass       |
+-----------------------+
| + templateMethod()    |  <-- 定义算法骨架
| # primitiveOperation1()| <-- 子类实现
| # primitiveOperation2()| <-- 子类实现
+-----------------------+
         ^
         |
+-----------------------+
|   ConcreteClass       |
+-----------------------+
| # primitiveOperation1()|
| # primitiveOperation2()|
+-----------------------+
```

### Spring AbstractApplicationContext.refresh()

Spring 的 `refresh()` 方法是模板方法模式的巅峰之作，定义了容器初始化的完整流程：

```java
public abstract class AbstractApplicationContext extends DefaultResourceLoader
        implements ConfigurableApplicationContext {

    public void refresh() throws BeansException, IllegalStateException {
        synchronized (this.startupShutdownMonitor) {
            prepareRefresh();                    // 1. 准备工作，记录状态

            ConfigurableListableBeanFactory beanFactory = obtainFreshBeanFactory(); // 2. 创建/获取 BeanFactory

            prepareBeanFactory(beanFactory);     // 3. 配置 BeanFactory

            try {
                postProcessBeanFactory(beanFactory);  // 4. 子类后处理（模板方法）
                invokeBeanFactoryPostProcessors(beanFactory); // 5. 调用 BeanFactoryPostProcessor
                registerBeanPostProcessors(beanFactory);      // 6. 注册 BeanPostProcessor
                initMessageSource();              // 7. 初始化国际化
                initApplicationEventMulticaster(); // 8. 初始化事件广播器
                onRefresh();                      // 9. 子类特殊初始化（模板方法）
                registerListeners();              // 10. 注册监听器
                finishBeanFactoryInitialization(beanFactory); // 11. 完成单例 Bean 初始化
                finishRefresh();                  // 12. 完成刷新
            } catch (BeansException ex) {
                destroyBeans();                   // 异常清理
                cancelRefresh(ex);
                throw ex;
            }
        }
    }

    // 子类可覆写的模板方法
    protected void postProcessBeanFactory(ConfigurableListableBeanFactory beanFactory) {
        // 默认空实现，子类可覆写
    }

    protected void onRefresh() throws BeansException {
        // 默认空实现，子类可覆写
    }
}
```

子类 `AnnotationConfigServletWebServerApplicationContext` 覆写了 `onRefresh()` 来创建内嵌 Web 服务器。

### Spring JdbcTemplate

`JdbcTemplate` 使用回调方式的模板方法（不使用继承，而使用组合 + 回调）：

```java
public class JdbcTemplate extends JdbcAccessor implements JdbcOperations {

    // 查询的模板方法
    public <T> T query(PreparedStatementCreator psc, ResultSetExtractor<T> rse) {
        // 1. 获取连接
        // 2. 创建 Statement
        // 3. 执行查询
        // 4. 处理结果集（回调）
        // 5. 关闭资源
        // 6. 异常转换
    }

    // 使用示例
    public List<User> findAllUsers() {
        return jdbcTemplate.query("SELECT * FROM users", (rs, rowNum) -> {
            User user = new User();
            user.setId(rs.getLong("id"));
            user.setName(rs.getString("name"));
            return user;
        });
    }
}
```

**回调方式的模板方法 vs 继承方式的模板方法**：

| 特性 | 继承方式 | 回调方式 |
|------|---------|---------|
| 扩展方式 | 继承抽象类 | 传入 Lambda/接口 |
| 灵活性 | 较低 | 较高 |
| 组合性 | 单继承限制 | 可自由组合 |
| 典型应用 | `AbstractApplicationContext` | `JdbcTemplate` |

## 观察者模式

### 概念

观察者模式定义对象间一对多的依赖关系，当一个对象状态改变时，所有依赖它的对象都会收到通知并自动更新。

### Spring Event 机制

Spring 的事件机制是观察者模式的完整实现，由三个核心组件构成：

```java
// 1. 定义事件
public class OrderCreatedEvent extends ApplicationEvent {
    private final Order order;

    public OrderCreatedEvent(Object source, Order order) {
        super(source);
        this.order = order;
    }

    public Order getOrder() {
        return order;
    }
}

// 2. 发布事件
@Service
public class OrderService {
    @Autowired
    private ApplicationEventPublisher eventPublisher;

    @Transactional
    public void createOrder(CreateOrderRequest request) {
        Order order = saveOrder(request);
        // 发布事件，所有监听器都会收到通知
        eventPublisher.publishEvent(new OrderCreatedEvent(this, order));
    }
}

// 3. 监听事件（观察者）
@Component
public class EmailNotificationListener {

    @Async // 异步处理
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        Order order = event.getOrder();
        sendConfirmationEmail(order.getUserEmail(), order.getOrderId());
    }
}

@Component
public class InventoryUpdateListener {

    @EventListener
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT) // 事务提交后执行
    public void onOrderCreated(OrderCreatedEvent event) {
        deductInventory(event.getOrder().getItems());
    }
}
```

**Spring Event 异步与事务**：

- `@EventListener`：默认同步执行，与发布者在同一线程
- `@Async + @EventListener`：异步执行，需要 `@EnableAsync` 支持
- `@TransactionalEventListener`：在事务特定阶段执行（BEFORE_COMMIT、AFTER_COMMIT 等）

### Vue 响应式

Vue 的响应式系统本质上也是观察者模式的变体：

```
数据变化 (Subject) -> 依赖收集 (Dep) -> 通知更新 (Watcher) -> 重新渲染视图
```

```javascript
// Vue 3 响应式原理简化
const targetMap = new WeakMap(); // 存储依赖关系

function reactive(obj) {
    return new Proxy(obj, {
        get(target, key) {
            track(target, key);  // 依赖收集：记录当前 Watcher
            return target[key];
        },
        set(target, key, value) {
            target[key] = value;
            trigger(target, key); // 触发更新：通知所有 Watcher
            return true;
        }
    });
}
```

## 责任链模式

### 概念

责任链模式将请求的发送者和接收者解耦，使多个对象都有机会处理请求。将这些对象连成一条链，沿着链传递请求，直到有一个对象处理为止。

### UML 类图

```
+------------------+     +------------------+
|   Handler        |     |   Request        |
+------------------+     +------------------+
| - next: Handler  |---->| + handle()       |
| + setNext()      |     +------------------+
| + handleRequest()|
+------------------+
```

### Servlet Filter

Servlet Filter 是责任链模式的经典应用，每个 Filter 可以在请求到达 Servlet 之前或之后执行逻辑：

```java
@Component
public class AuthFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        HttpServletRequest req = (HttpServletRequest) request;
        String token = req.getHeader("Authorization");

        if (token == null || !validateToken(token)) {
            ((HttpServletResponse) response).sendError(401, "未授权");
            return; // 不继续传递
        }

        // 前置处理
        System.out.println("认证通过");
        chain.doFilter(request, response); // 传递给下一个 Filter
        // 后置处理
        System.out.println("认证 Filter 后置处理");
    }
}

@Component
public class LogFilter implements Filter {
    @Override
    public void doFilter(ServletRequest request, ServletResponse response,
                         FilterChain chain) throws IOException, ServletException {
        long start = System.currentTimeMillis();
        System.out.println("请求开始: " + ((HttpServletRequest) request).getRequestURI());
        chain.doFilter(request, response);
        System.out.println("请求耗时: " + (System.currentTimeMillis() - start) + "ms");
    }
}

// 执行顺序：AuthFilter -> LogFilter -> Servlet -> LogFilter 后置 -> AuthFilter 后置
```

### Spring Interceptor

Spring MVC 的 `HandlerInterceptor` 也是责任链模式：

```java
@Component
public class RateLimitInterceptor implements HandlerInterceptor {
    private final RateLimiter limiter = RateLimiter.create(100); // 100 QPS

    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) {
        if (!limiter.tryAcquire()) {
            response.setStatus(429);
            return false; // 终止链
        }
        return true; // 继续传递
    }
}

@Component
public class TraceInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response,
                             Object handler) {
        String traceId = UUID.randomUUID().toString();
        MDC.put("traceId", traceId);
        request.setAttribute("traceId", traceId);
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request, HttpServletResponse response,
                                Object handler, Exception ex) {
        MDC.clear();
    }
}

// 注册拦截器
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Autowired
    private RateLimitInterceptor rateLimitInterceptor;
    @Autowired
    private TraceInterceptor traceInterceptor;

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(traceInterceptor)
                .addPathPatterns("/**")
                .order(1);
        registry.addInterceptor(rateLimitInterceptor)
                .addPathPatterns("/api/**")
                .order(2);
    }
}
```

**Filter vs Interceptor 对比**：

| 特性 | Filter | HandlerInterceptor |
|------|--------|-------------------|
| 规范 | Servlet 规范 | Spring 框架 |
| 作用范围 | 所有请求（包括静态资源） | 仅 Handler 请求 |
| 是否能获取 Handler | 否 | 是 |
| 注入 Bean | 需要额外配置 | 天然支持 |
| 执行时机 | 在 DispatcherServlet 前/后 | 在 Handler 执行前/后 |

## 命令模式

### 概念

命令模式将请求封装为对象，从而允许用不同的请求对客户端进行参数化、排队请求、记录日志，以及支持撤销操作。

### Runnable

Java 的 `Runnable` 接口就是命令模式的体现：

```java
// 命令接口
@FunctionalInterface
public interface Runnable {
    void run(); // 执行命令
}

// 具体命令
Runnable task1 = () -> System.out.println("发送邮件");
Runnable task2 = () -> System.out.println("生成报表");

// 调用者：线程池
ExecutorService executor = Executors.newFixedThreadPool(4);
executor.submit(task1); // 提交命令
executor.submit(task2);
```

### Spring JdbcTemplate 操作

JdbcTemplate 的 `execute` 方法也体现了命令模式，将数据库操作封装为 `ConnectionCallback` 或 `StatementCallback` 命令对象：

```java
// 命令接口
public interface ConnectionCallback<T> {
    T doInConnection(Connection con) throws SQLException, DataAccessException;
}

// 使用命令模式执行复杂操作
jdbcTemplate.execute(new ConnectionCallback<Void>() {
    @Override
    public Void doInConnection(Connection conn) throws SQLException {
        // 批量操作
        Statement stmt = conn.createStatement();
        stmt.addBatch("INSERT INTO users VALUES (1, '张三')");
        stmt.addBatch("INSERT INTO users VALUES (2, '李四')");
        stmt.executeBatch();
        return null;
    }
});
```

## 状态模式

### 概念

状态模式允许对象在内部状态改变时改变其行为，对象看起来像是修改了其类。将状态相关的行为封装到独立的状态类中。

### 实战场景：订单状态流转

```java
// 状态接口
public interface OrderState {
    void pay(OrderContext context);
    void ship(OrderContext context);
    void deliver(OrderContext context);
    void cancel(OrderContext context);
    String getStateName();
}

// 待支付状态
public class PendingPayState implements OrderState {
    @Override
    public void pay(OrderContext context) {
        System.out.println("支付成功");
        context.setState(new PaidState());
    }

    @Override
    public void ship(OrderContext context) {
        throw new IllegalStateException("待支付订单不能发货");
    }

    @Override
    public void deliver(OrderContext context) {
        throw new IllegalStateException("待支付订单不能确认收货");
    }

    @Override
    public void cancel(OrderContext context) {
        System.out.println("订单已取消");
        context.setState(new CancelledState());
    }

    @Override
    public String getStateName() { return "待支付"; }
}

// 已支付状态
public class PaidState implements OrderState {
    @Override
    public void pay(OrderContext context) {
        throw new IllegalStateException("订单已支付");
    }

    @Override
    public void ship(OrderContext context) {
        System.out.println("订单已发货");
        context.setState(new ShippedState());
    }

    @Override
    public void deliver(OrderContext context) {
        throw new IllegalStateException("已支付订单不能直接确认收货");
    }

    @Override
    public void cancel(OrderContext context) {
        System.out.println("退款成功，订单已取消");
        context.setState(new CancelledState());
    }

    @Override
    public String getStateName() { return "已支付"; }
}

// 上下文
public class OrderContext {
    private OrderState state;

    public OrderContext() {
        this.state = new PendingPayState();
    }

    public void setState(OrderState state) {
        this.state = state;
    }

    public void pay() { state.pay(this); }
    public void ship() { state.ship(this); }
    public void deliver() { state.deliver(this); }
    public void cancel() { state.cancel(this); }
    public String getState() { return state.getStateName(); }
}
```

**状态模式 vs 策略模式**：

| 特性 | 状态模式 | 策略模式 |
|------|---------|---------|
| 状态转换 | 状态对象内部管理 | 客户端控制 |
| 客户端感知 | 不感知状态变化 | 需要选择策略 |
| 行为约束 | 当前状态限制行为 | 策略之间独立 |
| 典型场景 | 工作流、状态机 | 算法选择 |

## 迭代器模式

### 概念

迭代器模式提供一种顺序访问聚合对象中元素的方法，而不暴露其底层表示。将遍历逻辑从聚合对象中分离出来。

### Java Iterator

```java
// Java Iterator 接口
public interface Iterator<E> {
    boolean hasNext();
    E next();
    default void remove() {
        throw new UnsupportedOperationException("remove");
    }
}

// ArrayList 的迭代器实现
public class ArrayList<E> extends AbstractList<E> {
    private Object[] elementData;
    private int size;

    public Iterator<E> iterator() {
        return new Itr();
    }

    private class Itr implements Iterator<E> {
        int cursor;       // 下一个元素的索引
        int lastRet = -1; // 上一个返回的元素索引
        int expectedModCount = modCount; // 快速失败机制

        public boolean hasNext() {
            return cursor != size;
        }

        @SuppressWarnings("unchecked")
        public E next() {
            checkForComodification();
            int i = cursor;
            if (i >= size) throw new NoSuchElementException();
            Object[] elementData = ArrayList.this.elementData;
            if (i >= elementData.length) throw new ConcurrentModificationException();
            cursor = i + 1;
            return (E) elementData[lastRet = i];
        }

        final void checkForComodification() {
            if (modCount != expectedModCount)
                throw new ConcurrentModificationException();
        }
    }
}
```

**快速失败（fail-fast）机制**：迭代器在创建时记录 `modCount`，每次调用 `next()` 时检查是否被修改。如果在迭代过程中通过非迭代器方法修改集合（如 `list.add()`），会抛出 `ConcurrentModificationException`。

### List 遍历方式对比

```java
List<String> list = Arrays.asList("Java", "Python", "Go");

// 1. for-each（语法糖，编译为迭代器）
for (String item : list) {
    System.out.println(item);
}

// 2. 显式迭代器（可安全 remove）
Iterator<String> it = list.iterator();
while (it.hasNext()) {
    String item = it.next();
    if ("Python".equals(item)) {
        it.remove(); // 安全删除
    }
}

// 3. forEach + Lambda（Java 8）
list.forEach(System.out::println);

// 4. ListIterator（支持双向遍历和修改）
ListIterator<String> lit = list.listIterator();
while (lit.hasNext()) {
    System.out.println(lit.nextIndex() + ": " + lit.next());
}
while (lit.hasPrevious()) {
    System.out.println(lit.previousIndex() + ": " + lit.previous());
}
```

## 总结

| 模式 | 核心意图 | 典型场景 | Spring 应用 |
|------|----------|----------|-------------|
| 策略 | 算法封装与替换 | 支付方式、排序策略 | ResourceLoader |
| 模板方法 | 定义算法骨架 | 框架流程控制 | refresh()、JdbcTemplate |
| 观察者 | 一对多通知 | 事件驱动 | Spring Event |
| 责任链 | 请求沿链传递 | 过滤、拦截 | Filter、Interceptor |
| 命令 | 请求封装为对象 | 异步执行、撤销 | JdbcTemplate |
| 状态 | 状态驱动的行为变化 | 订单流转、工作流 | - |
| 迭代器 | 顺序访问聚合对象 | 集合遍历 | Java Iterator |

行为型模式是日常开发中使用频率最高的一组模式。其中策略模式和模板方法模式在 Spring 框架中被大量使用，理解这两种模式有助于阅读和扩展 Spring 源码。观察者模式是实现事件驱动架构的基础，在微服务系统中常用于模块间解耦。
