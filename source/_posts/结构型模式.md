结构型设计模式关注类和对象的组合方式，通过继承和组合机制实现更灵活的结构。本文深入分析七种结构型模式，重点解析代理模式在 Spring AOP 和 RPC 框架中的核心应用。

## 代理模式

### 概念

代理模式为另一个对象提供替身或占位符，以控制对这个对象的访问。代理对象在客户端和目标对象之间起到中介作用，可以在不修改目标对象的前提下增强功能。

### UML 类图

```
+------------------+        +------------------+
|   Subject        |        |   RealSubject    |
+------------------+        +------------------+
| + request()      |<-------| + request()      |
+------------------+        +------------------+
        ^
        |
+------------------+
|   Proxy          |
+------------------+
| - realSubject    |
+------------------+
| + request()      |
+------------------+
```

### 静态代理

静态代理在编译时确定代理类，代理类和目标类实现相同接口。

```java
public interface UserDao {
    void save(User user);
    User findById(Long id);
}

public class UserDaoImpl implements UserDao {
    @Override
    public void save(User user) {
        System.out.println("保存用户: " + user.getName());
    }

    @Override
    public User findById(Long id) {
        System.out.println("查询用户: " + id);
        return new User(id, "张三");
    }
}

// 静态代理类
public class UserDaoProxy implements UserDao {
    private final UserDao target;

    public UserDaoProxy(UserDao target) {
        this.target = target;
    }

    @Override
    public void save(User user) {
        System.out.println("[LOG] 开始保存用户");
        long start = System.currentTimeMillis();
        target.save(user);
        long cost = System.currentTimeMillis() - start;
        System.out.println("[LOG] 保存完成，耗时: " + cost + "ms");
    }

    @Override
    public User findById(Long id) {
        System.out.println("[LOG] 开始查询用户");
        return target.findById(id);
    }
}
```

**缺点**：每个目标类都需要一个代理类，接口方法增加时代理类也要同步修改。

### JDK 动态代理

JDK 动态代理通过反射在运行时生成代理类，要求目标类必须实现接口。

```java
public class JdkProxyHandler implements InvocationHandler {
    private final Object target;

    public JdkProxyHandler(Object target) {
        this.target = target;
    }

    @SuppressWarnings("unchecked")
    public <T> T getProxy() {
        return (T) Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            target.getClass().getInterfaces(),
            this
        );
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[JDK Proxy] 前置通知: " + method.getName());
        long start = System.currentTimeMillis();

        try {
            Object result = method.invoke(target, args);
            System.out.println("[JDK Proxy] 返回通知，耗时: "
                + (System.currentTimeMillis() - start) + "ms");
            return result;
        } catch (Exception e) {
            System.out.println("[JDK Proxy] 异常通知: " + e.getMessage());
            throw e;
        } finally {
            System.out.println("[JDK Proxy] 后置通知");
        }
    }
}

// 使用方式
UserDao original = new UserDaoImpl();
UserDao proxy = new JdkProxyHandler(original).getProxy();
proxy.save(new User(1L, "李四"));
```

**原理**：`Proxy.newProxyInstance()` 在运行时生成一个实现了指定接口的代理类字节码，并加载到 JVM 中。代理类将方法调用转发给 `InvocationHandler.invoke()` 方法。

### CGLIB 动态代理

CGLIB 通过字节码技术生成目标类的子类来实现代理，不要求目标类实现接口。

```java
public class CglibProxyInterceptor implements MethodInterceptor {
    @Override
    public Object intercept(Object obj, Method method, Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("[CGLIB] 前置增强: " + method.getName());
        long start = System.currentTimeMillis();

        try {
            // 通过 FastClass 机制调用父类方法（比反射高效）
            Object result = proxy.invokeSuper(obj, args);
            System.out.println("[CGLIB] 返回增强，耗时: "
                + (System.currentTimeMillis() - start) + "ms");
            return result;
        } catch (Exception e) {
            System.out.println("[CGLIB] 异常增强: " + e.getMessage());
            throw e;
        }
    }
}

// 使用方式
Enhancer enhancer = new Enhancer();
enhancer.setSuperclass(UserDaoImpl.class);
enhancer.setCallback(new CglibProxyInterceptor());
UserDaoImpl proxy = (UserDaoImpl) enhancer.create();
proxy.save(new User(1L, "王五"));
```

**CGLIB vs JDK 动态代理对比**：

| 特性 | JDK 动态代理 | CGLIB |
|------|-------------|-------|
| 要求 | 目标类实现接口 | 目标类可被继承 |
| 原理 | 反射调用 | 生成子类 + FastClass |
| 性能 | 生成快，调用稍慢 | 生成慢，调用更快 |
| 限制 | 无法代理 final 方法 | 无法代理 final 类/方法 |

### Spring AOP 中的应用

Spring AOP 默认使用 JDK 动态代理（当 Bean 实现了接口时），否则使用 CGLIB。从 Spring Boot 2.x 开始，默认使用 CGLIB。

```java
// 自定义切面
@Aspect
@Component
public class LoggingAspect {

    // 切点：匹配 Service 层所有方法
    @Pointcut("execution(* com.example.service..*.*(..))")
    public void serviceLayer() {}

    @Before("serviceLayer()")
    public void beforeAdvice(JoinPoint joinPoint) {
        String methodName = joinPoint.getSignature().getName();
        Object[] args = joinPoint.getArgs();
        System.out.println("[AOP] 调用方法: " + methodName + ", 参数: " + Arrays.toString(args));
    }

    @AfterReturning(pointcut = "serviceLayer()", returning = "result")
    public void afterReturningAdvice(JoinPoint joinPoint, Object result) {
        System.out.println("[AOP] 方法返回: " + result);
    }

    @Around("serviceLayer()")
    public Object aroundAdvice(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed();
        long cost = System.currentTimeMillis() - start;
        System.out.println("[AOP] " + pjp.getSignature().getName() + " 耗时: " + cost + "ms");
        return result;
    }
}
```

**Spring AOP 代理创建流程**：

```
AbstractAutoProxyCreator.postProcessAfterInitialization()
    -> wrapIfNecessary()
        -> getAdvicesAndAdvisorsForInterface() // 获取切面
        -> createProxy()                       // 创建代理
            -> 如果目标类实现接口且配置为 JDK 代理
                -> JdkDynamicAopProxy
            -> 否则
                -> CglibAopProxy
```

在 RPC 框架（如 Dubbo、Feign）中，消费者端的接口实际上就是代理对象。调用远程方法就像调用本地方法一样，代理对象负责序列化请求、网络传输、反序列化响应。

## 适配器模式

### 概念

适配器模式将一个类的接口转换成客户端期望的另一种接口，使原本不兼容的类可以协同工作。分为类适配器（继承）和对象适配器（组合）两种。

### 代码示例

```java
// 旧系统的日志接口
public interface LegacyLogger {
    void logMessage(String level, String message);
}

// 新系统的日志接口
public interface NewLogger {
    void info(String message);
    void error(String message);
}

// 对象适配器：让旧的日志接口适配新接口
public class LoggerAdapter implements NewLogger {
    private final LegacyLogger legacyLogger;

    public LoggerAdapter(LegacyLogger legacyLogger) {
        this.legacyLogger = legacyLogger;
    }

    @Override
    public void info(String message) {
        legacyLogger.logMessage("INFO", message);
    }

    @Override
    public void error(String message) {
        legacyLogger.logMessage("ERROR", message);
    }
}
```

### SpringMVC HandlerAdapter

SpringMVC 的 `HandlerAdapter` 是适配器模式的经典应用。不同类型的 Handler（Controller、Servlet、HttpRequestHandler）具有不同的调用方式，`HandlerAdapter` 统一了调用接口。

```java
// DispatcherServlet 核心分发逻辑
public class DispatcherServlet {
    private List<HandlerAdapter> handlerAdapters;

    protected void doDispatch(HttpServletRequest request, HttpServletResponse response) {
        Object handler = getHandler(request);
        // 根据 handler 找到对应的适配器
        HandlerAdapter adapter = getHandlerAdapter(handler);
        // 统一调用
        ModelAndView mv = adapter.handle(request, response, handler);
    }

    protected HandlerAdapter getHandlerAdapter(Object handler) {
        for (HandlerAdapter adapter : handlerAdapters) {
            if (adapter.supports(handler)) {
                return adapter;
            }
        }
        throw new ServletException("No adapter for handler: " + handler);
    }
}

// 三种适配器实现
// 1. RequestMappingHandlerAdapter -> @RequestMapping 方法
// 2. HttpRequestHandlerAdapter -> HttpRequestHandler 接口
// 3. SimpleControllerHandlerAdapter -> Controller 接口
```

这种设计使得 `DispatcherServlet` 无需关心具体 Handler 的类型，只通过 `HandlerAdapter` 统一调用，新增 Handler 类型时只需添加对应的 Adapter 即可。

## 装饰器模式

### 概念

装饰器模式动态地给对象增加功能，是继承的一种替代方案。装饰器和被装饰者实现相同接口，通过层层包装来扩展功能。

### UML 类图

```
+------------------+        +------------------+
|   Component      |        |   Decorator      |
+------------------+        +------------------+
| + operation()    |<-------| - component      |
+------------------+        | + operation()    |
        ^                   +------------------+
        |                           ^
        |                           |
+------------------+        +------------------+
| ConcreteComponent|        | ConcreteDecorator|
+------------------+        +------------------+
| + operation()    |        | + operation()    |
+------------------+        | + addedBehavior()|
                            +------------------+
```

### Java IO 流体系

Java IO 是装饰器模式最经典的体现：

```java
// 装饰器层层包装
InputStream input = new BufferedInputStream(             // 缓冲装饰器
                        new GZIPInputStream(             // 解压缩装饰器
                            new FileInputStream("data.gz") // 基础节点流
                        )
                    );

// 读取过程：BufferedInputStream -> GZIPInputStream -> FileInputStream
// 每一层都增加了额外功能（缓冲 / 解压 / 文件读取）
```

**IO 流中的角色映射**：

| 角色 | 对应的类 |
|------|---------|
| Component | `InputStream` / `OutputStream` |
| ConcreteComponent | `FileInputStream` / `ByteArrayInputStream` |
| Decorator | `FilterInputStream` / `FilterOutputStream` |
| ConcreteDecorator | `BufferedInputStream` / `GZIPInputStream` |

核心源码分析：

```java
// FilterInputStream 是所有装饰器的基类
public class FilterInputStream extends InputStream {
    protected volatile InputStream in; // 持有被装饰对象

    public int read() throws IOException {
        return in.read(); // 默认直接委托，子类覆写以增加功能
    }
}

// BufferedInputStream 增加缓冲功能
public class BufferedInputStream extends FilterInputStream {
    protected volatile byte[] buf; // 内部缓冲区

    public synchronized int read() throws IOException {
        // 如果缓冲区为空，批量读取数据到缓冲区
        // 然后从缓冲区返回单个字节
    }
}
```

## 桥接模式

### 概念

桥接模式将抽象部分与实现部分分离，使它们都可以独立变化。通过组合而非继承来连接两个维度。

### JDBC Driver 中的桥接

JDBC 是桥接模式的经典应用。Java 应用通过 JDBC API（抽象部分）操作数据库，具体的数据库驱动（实现部分）可以独立切换。

```java
// 抽象部分：JDBC API
Connection conn = DriverManager.getConnection(url, user, password);
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery("SELECT * FROM users");

// 切换实现部分只需更改驱动和 URL
// MySQL
Class.forName("com.mysql.cj.jdbc.Driver");
Connection mysqlConn = DriverManager.getConnection("jdbc:mysql://localhost/db", ...);

// PostgreSQL
Class.forName("org.postgresql.Driver");
Connection pgConn = DriverManager.getConnection("jdbc:postgresql://localhost/db", ...);

// 应用代码完全不变
```

**桥接结构**：
- 抽象层：`java.sql.Connection`、`java.sql.Statement`、`java.sql.ResultSet`
- 实现层：`com.mysql.cj.jdbc.ConnectionImpl`、`org.postgresql.jdbc.PgConnection`
- 桥接点：`DriverManager` 通过驱动接口连接两者

## 组合模式

### 概念

组合模式将对象组合成树形结构，使客户端对单个对象和组合对象的使用具有一致性。适用于部分-整体的层次结构。

### Spring Cache 缓存层级

Spring Cache 的 `CompositeCacheManager` 组合多个 CacheManager，形成缓存层级：

```java
// CompositeCacheManager 持有多个 CacheManager
public class CompositeCacheManager implements CacheManager {
    private final List<CacheManager> cacheManagers;

    @Override
    public Cache getCache(String name) {
        // 按顺序从每个 CacheManager 中查找
        for (CacheManager cacheManager : cacheManagers) {
            Cache cache = cacheManager.getCache(name);
            if (cache != null) {
                return cache;
            }
        }
        return null;
    }
}

// 配置多级缓存
@Configuration
@EnableCaching
public class CacheConfig {
    @Bean
    public CompositeCacheManager cacheManager() {
        CompositeCacheManager composite = new CompositeCacheManager();
        composite.setCacheManagers(
            Arrays.asList(
                caffeineCacheManager(),   // 本地缓存（L1）
                redisCacheManager()       // 分布式缓存（L2）
            )
        );
        return composite;
    }
}
```

## 外观模式

### 概念

外观模式为子系统中的一组接口提供一个统一的高层接口，简化子系统的使用难度。客户端不需要了解子系统的复杂内部结构。

### Spring JdbcTemplate

`JdbcTemplate` 是外观模式的典型应用，封装了 JDBC 的复杂操作流程：

```java
// 不使用 JdbcTemplate 的原始 JDBC 操作
public User findById_raw(Long id) {
    Connection conn = null;
    PreparedStatement ps = null;
    ResultSet rs = null;
    try {
        conn = dataSource.getConnection();
        ps = conn.prepareStatement("SELECT * FROM users WHERE id = ?");
        ps.setLong(1, id);
        rs = ps.executeQuery();
        if (rs.next()) {
            User user = new User();
            user.setId(rs.getLong("id"));
            user.setName(rs.getString("name"));
            return user;
        }
    } catch (SQLException e) {
        throw new RuntimeException(e);
    } finally {
        // 关闭 rs、ps、conn（每个都要 try-catch）
        if (rs != null) try { rs.close(); } catch (SQLException e) {}
        if (ps != null) try { ps.close(); } catch (SQLException e) {}
        if (conn != null) try { conn.close(); } catch (SQLException e) {}
    }
    return null;
}

// 使用 JdbcTemplate（外观模式）
public User findById(Long id) {
    return jdbcTemplate.queryForObject(
        "SELECT * FROM users WHERE id = ?",
        (rs, rowNum) -> {
            User user = new User();
            user.setId(rs.getLong("id"));
            user.setName(rs.getString("name"));
            return user;
        },
        id
    );
}
```

`JdbcTemplate` 内部封装了连接获取、Statement 创建、异常转换（`SQLException` -> `DataAccessException`）、资源关闭等所有样板代码。

## 享元模式

### 概念

享元模式通过共享对象来最小化内存使用，适合大量相似对象的场景。分离内部状态（可共享）和外部状态（不可共享）。

### Integer Cache

Java `Integer` 对 -128 到 127 的整数使用了享元模式缓存：

```java
public final class Integer extends Number implements Comparable<Integer> {
    // 缓存 -128 到 127 的 Integer 对象
    private static class IntegerCache {
        static final int low = -128;
        static final int high; // 默认 127，可通过 -XX:AutoBoxCacheMax 调整
        static final Integer[] cache;

        static {
            int h = 127;
            cache = new Integer[(high = h) - (low) + 1];
            int j = low;
            for (int i = 0; i < cache.length; i++) {
                cache[i] = new Integer(j++);
            }
        }
    }

    public static Integer valueOf(int i) {
        if (i >= IntegerCache.low && i <= IntegerCache.high) {
            return IntegerCache.cache[i + (-IntegerCache.low)];
        }
        return new Integer(i);
    }
}
```

这解释了一个常见的面试题：

```java
Integer a = 127;
Integer b = 127;
System.out.println(a == b);  // true，共享同一个缓存对象

Integer c = 128;
Integer d = 128;
System.out.println(c == d);  // false，超出缓存范围，各自 new
```

### String 常量池

String 常量池也是享元模式的体现：

```java
String s1 = "hello";         // 放入常量池
String s2 = "hello";         // 复用常量池中的对象
System.out.println(s1 == s2); // true

String s3 = new String("hello"); // 堆上创建新对象
System.out.println(s1 == s3);    // false

String s4 = s3.intern();         // 尝试放入常量池
System.out.println(s1 == s4);    // true
```

JDK 7 之后，String 常量池从永久代移到了堆中，`intern()` 方法如果常量池中已有等值字符串则返回引用，否则将堆中字符串的引用存入常量池（而非复制字符串本身）。

## 总结

| 模式 | 核心意图 | 典型场景 | Spring/Java 应用 |
|------|----------|----------|-----------------|
| 代理 | 控制访问，增强功能 | AOP、RPC、懒加载 | Spring AOP、Feign |
| 适配器 | 接口转换 | 系统集成、兼容旧接口 | HandlerAdapter |
| 装饰器 | 动态增强 | IO 流、功能叠加 | Java IO |
| 桥接 | 抽象与实现分离 | 多维度变化 | JDBC Driver |
| 组合 | 树形结构统一处理 | 层级结构、多级缓存 | CompositeCacheManager |
| 外观 | 简化接口 | 封装复杂子系统 | JdbcTemplate |
| 享元 | 共享对象减少内存 | 大量相似对象 | Integer Cache、String Pool |

结构型模式的共同目标是**通过合理的结构设计减少系统的复杂度**。代理模式在 Spring 生态中应用最为广泛，理解 JDK 动态代理和 CGLIB 的原理对于排查 AOP 相关问题至关重要。
